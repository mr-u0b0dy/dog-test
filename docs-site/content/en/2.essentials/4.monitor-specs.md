# Monitor Specs

A **monitor spec** tells the host runner to capture logic-analyser data during
test execution and validate the captured signals against expectations.

## Defining a Monitor Spec (C)

```c
#include "dogtest/monitor_spec.h"

static const dt_monitor_spec_t my_uart_monitor = {
    "uart",                    // protocol
    "ch0=tx,ch1=rx",          // channel mapping
    "start-on-byte:0x55",     // trigger condition
    2000                       // timeout in ms
};

DT_TEST("uart_test", DT_RESET_SOFT, &my_uart_monitor, uart_test) {
    // ... test body ...
    return DT_TEST_PASSED;
}
```

Convenience macros are also available:

```c
DT_TEST("spi_test", DT_RESET_SOFT,
        DT_MONITOR_SPI("ch0=clk,ch1=mosi,ch2=miso,ch3=cs", "start-on-cs", 1000),
        spi_test) { ... }
```

## JSON Monitor Spec (Host)

Pass a monitor spec directly via CLI:

```bash
python3 tools/dt_runner.py \
  --firmware build/... \
  --monitor-spec '{
    "protocol": "uart",
    "channels": "ch0=tx,ch1=rx",
    "trigger": "start-on-byte:0x55",
    "timeout_ms": 2000,
    "expect": {
      "protocol": "uart",
      "min_items": 1,
      "contains": "OK"
    }
  }'
```

## Expect Schema

The `expect` object supports three keys:

| Key | Type | Description |
|-----|------|-------------|
| `protocol` | `string` | Assert the captured protocol matches |
| `min_items` | `int` | Minimum number of decoded frames/transactions |
| `contains` | `string` | Assert decoded output contains this substring |

Unknown keys in `expect` will raise a `ValueError` at parse time to catch
typos early (e.g. `min_item` instead of `min_items`).

## Test Plans with Monitors

In a JSON test plan, monitors can be inline or file references:

```json
{
  "tests": [
    {
      "name": "uart_loopback",
      "firmware": "build/examples/uart_loopback/uart_loopback",
      "monitor": "monitor_spec_uart.json"
    }
  ]
}
```

File paths are resolved relative to the plan file's directory.
