# Examples

dog-test ships with three example firmware tests and two JSON spec files. Each example demonstrates a different protocol and reset mode.

## UART Loopback

**Location:** `examples/uart_loopback/`

A minimal UART test that verifies a sent byte matches the received byte (loopback). Uses `HT_RESET_NONE` (no reset before test) and attaches a Saleae monitor spec.

```c [examples/uart_loopback/main.c]
#include <stdio.h>
#include "hiltest/assert.h"
#include "hiltest/monitor_spec.h"
#include "hiltest/test_case.h"

static const ht_monitor_spec_t uart_monitor = {
    "uart",
    "ch0=tx,ch1=rx",
    "start-on-byte:0x55",
    1000
};

HT_TEST("uart_loopback_basic", HT_RESET_NONE, &uart_monitor, uart_loopback_basic) {
    int sent = 0x55;
    int received = 0x55;
    HT_ASSERT_EQ(sent, received);
    HT_ASSERT_TRUE(received == 0x55);
    return HT_TEST_PASSED;
}

HT_MAIN()
```

**What to notice:**
- `ht_monitor_spec_t` describes the expected Saleae capture: protocol, channel mapping, trigger condition, and timeout.
- `HT_RESET_NONE` — the target is not reset before this test.
- On real hardware, `sent` and `received` would come from actual UART peripheral registers.

## I2C Smoke

**Location:** `examples/i2c_smoke/`

Verifies that an I2C device responds with the expected address. Uses `HT_RESET_SOFT`.

```c [examples/i2c_smoke/main.c]
static const ht_monitor_spec_t i2c_monitor = {
    "i2c",
    "ch0=scl,ch1=sda",
    "address:0x48",
    1200
};

HT_TEST("i2c_device_id", HT_RESET_SOFT, &i2c_monitor, i2c_device_id) {
    int expected = 0x48;
    int observed = 0x48;
    HT_ASSERT_EQ(expected, observed);
    return HT_TEST_PASSED;
}
```

**What to notice:**
- `HT_RESET_SOFT` — a software reset is issued before the test.
- The monitor spec triggers on the I2C address `0x48`.

## SPI Transfer

**Location:** `examples/spi_transfer/`

Validates a CRC check after an SPI transfer. Uses `HT_RESET_HARD` and a 4-channel Saleae capture.

```c [examples/spi_transfer/main.c]
static const ht_monitor_spec_t spi_monitor = {
    "spi",
    "ch0=clk,ch1=mosi,ch2=miso,ch3=cs",
    "cs-falling",
    1200
};

HT_TEST("spi_transfer_crc", HT_RESET_HARD, &spi_monitor, spi_transfer_crc) {
    int expected_crc = 0xAB;
    int actual_crc = 0xAB;
    HT_ASSERT_EQ(expected_crc, actual_crc);
    HT_ASSERT_NE(0x00, actual_crc);
    return HT_TEST_PASSED;
}
```

**What to notice:**
- `HT_RESET_HARD` — a hardware reset (power cycle or NRST) before the test.
- 4-channel capture: clock, MOSI, MISO, and chip-select.
- Two assertions: the CRC matches AND is non-zero.

## Monitor Spec (JSON)

**Location:** `examples/monitor_spec_uart.json`

A standalone monitor spec file that can be referenced from a test plan:

```json [examples/monitor_spec_uart.json]
{
  "protocol": "uart",
  "channels": "ch0=tx,ch1=rx",
  "trigger": "start-on-byte:0x55",
  "timeout_ms": 1200,
  "expect": {
    "protocol": "uart",
    "min_items": 1,
    "contains": "0x55"
  }
}
```

The `expect` block is evaluated by the host after capture. This one asserts that at least one UART frame containing `0x55` was captured.

## Test Plan (JSON)

**Location:** `tests/test_plan_smoke.json`

A test plan that runs all three examples in sequence:

```json [tests/test_plan_smoke.json]
{
  "tests": [
    {
      "name": "uart_loopback_once",
      "firmware": "build/host-debug/examples/uart_loopback/uart_loopback",
      "reset_mode": "none"
    },
    {
      "name": "i2c_smoke_once",
      "firmware": "build/host-debug/examples/i2c_smoke/i2c_smoke",
      "reset_mode": "soft"
    },
    {
      "name": "spi_transfer_once",
      "firmware": "build/host-debug/examples/spi_transfer/spi_transfer",
      "reset_mode": "hard"
    }
  ]
}
```

Run it with:

```bash [Terminal]
python3 tools/hil_runner.py --test-plan tests/test_plan_smoke.json --backend pyocd
```
