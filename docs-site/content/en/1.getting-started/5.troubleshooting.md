## Flash / Reset Failures

### pyOCD: "No connected debug probes"

Your debug probe is not detected. Check:

1. The USB cable is connected and the probe is powered.
2. `pyocd list` shows your probe.
3. Your user has permission to access USB devices (on Linux, check udev rules).

### OpenOCD: "Error: unable to open ftdi device"

Same root cause — the probe is not visible. Additionally:

- Ensure OpenOCD is built with the correct interface driver (FTDI, CMSIS-DAP, etc.).
- Try running with `sudo` to rule out permission issues, then fix udev rules.

### Flash succeeds but test times out

The firmware may be crashing before serial output begins. Check:

- The correct binary is being flashed (match the board's memory map).
- Clock and startup code initialise correctly.
- The `DT_MAIN()` macro is present and the test binary links `dogtest_core`.

## Serial Capture Issues

### No `DT_EVENT` lines received

- Verify the serial port path (`--port /dev/ttyACM0` or similar).
- Check the baud rate matches your firmware's UART configuration (default: 115200).
- Ensure UART TX on the target is connected to the host's RX.
- Try `minicom` or `picocom` manually to confirm data is arriving.

### Garbled or partial output

- Baud rate mismatch is the most common cause.
- Electrical noise on long wires — use shorter cables or add a ground connection.
- If using USB-CDC, ensure the host enumerates the device before opening the port (add a small delay or use `--retries`).

### `DT_EVENT` lines contain unexpected characters

Firmware `printf` output or debug logging may be interleaved with test events. The parser ignores lines that don't start with `DT_EVENT`, but interleaved output within a line can corrupt parsing. Route debug output to a different UART or disable it during tests.

## Build Issues

### CMake preset not found

```
CMake Error: Could not read presets from /path/to/dog-test
```

Ensure you're running from the project root and that `CMakePresets.json` exists. Check `cmake --list-presets` for available options.

### `arm-none-eabi-gcc` not found

The ARM cross-compiler is not on your `PATH`. Install it:

```bash [Terminal]
# Ubuntu/Debian
sudo apt install gcc-arm-none-eabi

# macOS (Homebrew)
brew install --cask gcc-arm-embedded

# Or download from https://developer.arm.com/tools-and-software/open-source-software/developer-tools/gnu-toolchain/gnu-rm
```

### Linker errors: undefined reference to `dt_*`

Your test binary isn't linking `dogtest_core`. In your `CMakeLists.txt`:

```cmake
target_link_libraries(my_test PRIVATE dogtest_core)
```

## Python Tooling

### `ModuleNotFoundError: No module named 'serial'`

Install pyserial:

```bash [Terminal]
pip install pyserial
```

Note: the package is `pyserial`, not `serial` (which is a different, incompatible package).

### `ModuleNotFoundError: No module named 'saleae'`

Saleae automation is optional. Install it only if you need logic analyser captures:

```bash [Terminal]
pip install saleae-automation
```

### pytest: tests not discovered

Ensure you're running from the project root:

```bash [Terminal]
python -m pytest tests/python/ -v
```

## Saleae Logic Integration

### "Unable to connect to Logic 2 software"

The `saleae-automation` SDK requires the Saleae Logic 2 desktop application to be running with the **Automation API** enabled:

1. Open Logic 2 → Preferences → Automation.
2. Enable "Enable automation server".
3. Note the port (default: 10430).

### No captures returned

- Ensure at least one logic analyser is connected.
- Check that the channels configured in your monitor spec match the physical connections.
- Verify the capture duration is long enough for the test to complete.
